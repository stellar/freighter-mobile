# Internationalization (i18n)

## Framework

The app uses `i18next` + `react-i18next`, initialized in `src/i18n/index.ts`.

## Supported Languages

| Language   | Code | Location                                |
| ---------- | ---- | --------------------------------------- |
| English    | `en` | `src/i18n/locales/en/translations.json` |
| Portuguese | `pt` | `src/i18n/locales/pt/translations.json` |

## Device Language Detection

`getDeviceLanguage()` detects the device's preferred language and falls back to
English if the detected language is not supported.

## Custom Hook

Prefer `useAppTranslation()` over the raw `useTranslation` hook for new code. It
wraps `useTranslation("translations")` so the typed translations namespace is
always selected. A handful of existing components (`TokenIcon.tsx`,
`InformationBottomSheet.tsx`, `AddMemoExplanationBottomSheet.tsx`,
`ContactRow.tsx`, `DappMessageDisplay.tsx`) still use the raw hook — don't flag
these as violations when touching them, but use `useAppTranslation` in any new
code you write.

```tsx
import useAppTranslation from "hooks/useAppTranslation";

const MyComponent: React.FC = () => {
  const { t } = useAppTranslation();

  return <Text>{t("home.title")}</Text>;
};
```

`useAppTranslation` is a **default export** — import it without curly braces.

## Key Structure

Translation keys use nested JSON objects with dot notation:

```json
{
  "transaction": {
    "errors": {
      "amountRequired": "Amount is required",
      "insufficientBalance": "Insufficient balance",
      "invalidDestination": "Invalid destination address"
    },
    "labels": {
      "amount": "Amount",
      "destination": "Destination"
    }
  }
}
```

Access in code: `t("transaction.errors.amountRequired")`

## Adding New Strings

1. Add the key to `src/i18n/locales/en/translations.json`
2. Add the same key to `src/i18n/locales/pt/translations.json` with the
   Portuguese translation
3. Use `t("your.new.key")` in the component

Both language files must be updated simultaneously. Missing translations will
trigger ESLint errors.

## ESLint Enforcement

The custom ESLint plugin at `src/eslint-plugin-translations/` enforces
translation usage:

- **Rule**: `translations/missing-translations` (error level)
- **What it does**: Best-effort detection of direct string literals in JSX that
  should use `t()` instead. The rule misses some cases — treat it as a safety
  net, not a guarantee. Every user-facing string must still come from `t()`
  calls.

```tsx
// Correct
<Text>{t("home.welcome")}</Text>
<Button title={t("common.submit")} />

// Wrong - will fail lint
<Text>Welcome</Text>
<Button title="Submit" />
```
